# Case 03: SSH login attempt with non-existent user

This writeup uses the **actual output** of `triage_agent.py` (Gemini `gemini-flash-lite-latest`), captured in `triage_log.jsonl`.

Unlike the other cases, this alert was **not** generated by my own scanner — it's real, unsolicited traffic from the public internet hitting the VM's exposed SSH port, captured while the scan was running.

## Raw Alert (as fed to the agent)

```json
{
  "_id": "sample-003",
  "@timestamp": "2026-07-18T18:34:32.000Z",
  "rule": {
    "id": "5710",
    "level": 5,
    "description": "sshd: Attempt to login using a non-existent user",
    "groups": ["syslog", "sshd", "authentication_failed", "invalid_login"],
    "mitre": { "id": ["T1110"] }
  },
  "location": "journald",
  "srcip": "165.22.178.87",
  "srcport": 60622,
  "full_log": "Jul 18 18:26:33 siem-target sshd[97650]: Invalid user admin from 165.22.178.87 port 60622"
}
```

## What Gemini actually returned

```json
{
  "summary": "An SSH login attempt was made using the non-existent username 'admin' from source IP 165.22.178.87.",
  "severity": "Low",
  "mitre_id": "T1110",
  "recommended_action": "Check if this source IP is associated with broader brute-force activity or other failed logins across the infrastructure; consider blocking the IP if it is unauthenticated external traffic.",
  "confidence": "High",
  "reasoning": "The alert rule level is 5 and the full_log explicitly shows an 'Invalid user admin' attempt from external source IP 165.22.178.87, which is a standard indicator of reconnaissance or automated credential guessing, but represents only a single failed attempt without evidence of compromise."
}
```

## Assessment

Solid, well-calibrated triage. The model correctly identified this as routine internet background scanning rather than a targeted or successful compromise, and — notably — its recommended action pushes toward exactly the right next step: checking whether this IP shows a **pattern** of repeated attempts across the infrastructure, rather than treating this single alert as the full picture.

## Verification against ground truth

No scanner ground truth applies here since this wasn't generated by my own tooling — it's real attacker/scanner traffic. The reasoning is validated by security best practice rather than a scanner comparison: SSH port-22 exposure on any public cloud IP reliably attracts this kind of traffic within minutes of going live, and a single non-existent-user attempt is standard, low-priority signal.

## Lesson

Not every fired alert represents a meaningful event requiring action. Some of the most valuable triage judgment is recognizing **ambient internet noise** and correctly not escalating it, while still logging it in case it becomes part of a larger pattern later — which is exactly the distinction the model's own recommended action pointed toward, without being explicitly told to.
